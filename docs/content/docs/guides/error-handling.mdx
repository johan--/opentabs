When a tool fails, how you report the error determines whether the AI agent retries intelligently, gives up, or tells the user something unhelpful. OpenTabs provides `ToolError` — a structured error class that carries machine-readable metadata through the entire dispatch chain to the AI agent.

## The Problem with Unstructured Errors

If your tool throws a plain `Error`, the AI agent sees a generic failure message with no context:

```
Tool dispatch error: Request failed
```

The agent doesn't know if this is temporary (retry might work) or permanent (don't bother). It doesn't know if the user needs to re-authenticate, or if the input was wrong. It guesses — and often guesses poorly.

## ToolError

`ToolError` adds three pieces of metadata to every error:

- **`category`** — what kind of failure (`auth`, `rate_limit`, `not_found`, `validation`, `timeout`, `internal`)
- **`retryable`** — whether retrying might succeed
- **`retryAfterMs`** — how long to wait before retrying (optional)

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

throw new ToolError('Channel not found', 'CHANNEL_NOT_FOUND', {
  category: 'not_found',
  retryable: false,
});
```

The AI agent receives both a human-readable message and a machine-readable JSON block:

```
[ERROR code=CHANNEL_NOT_FOUND category=not_found retryable=false] Channel not found

```json
{"code":"CHANNEL_NOT_FOUND","category":"not_found","retryable":false}
```
```

This tells the agent exactly what happened and what to do next.

## Factory Methods

Instead of constructing `ToolError` directly, use the static factory methods. They set the right category, code, and retryable flag automatically.

### `ToolError.auth(message)`

The user isn't authenticated, or the session expired. Not retryable — the user needs to log in.

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

async function handle() {
  const token = getLocalStorage('auth_token');
  if (!token) {
    throw ToolError.auth('Not logged in — open the app and sign in first');
  }
  // ...
}
```

### `ToolError.notFound(message, code?)`

The requested resource doesn't exist. Not retryable. Accepts an optional domain-specific code (defaults to `NOT_FOUND`).

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

async function handle({ channelName }: { channelName: string }) {
  const channel = findChannel(channelName);
  if (!channel) {
    throw ToolError.notFound(
      `Channel #${channelName} not found`,
      'CHANNEL_NOT_FOUND',
    );
  }
  // ...
}
```

### `ToolError.rateLimited(message, retryAfterMs?)`

The API is rate limiting requests. Retryable — the agent should wait and try again.

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

async function handle() {
  const response = await fetch('/api/search', { credentials: 'include' });
  if (response.status === 429) {
    const retryAfter = response.headers.get('Retry-After');
    const delayMs = retryAfter ? parseInt(retryAfter, 10) * 1000 : 5000;
    throw ToolError.rateLimited('Too many requests — try again shortly', delayMs);
  }
  // ...
}
```

### `ToolError.validation(message)`

  }

### `ToolError.timeout(message)`

The operation took too long. Retryable — transient network or server slowness.

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

async function handle() {
  try {
    const data = await fetchJSON('/api/report', {
      signal: AbortSignal.timeout(10_000),
    });
    return data;
  } catch (err) {
    if (err instanceof DOMException && err.name === 'TimeoutError') {
      throw ToolError.timeout('Report generation timed out after 10 seconds');
    }
    throw err;
  }
}
```

## How Errors Reach the AI Agent

When your tool throws a `ToolError`, it travels through the platform:

1. **Tool handler** throws `ToolError` in the page context
2. **Browser extension** serializes the error, preserving `code`, `category`, `retryable`, and `retryAfterMs`
3. **MCP server** formats the error into a human-readable prefix line and a machine-readable JSON block
4. **AI agent** receives both formats and uses the structured fields to decide what to do next

The AI agent sees a response like this:

```
[ERROR code=RATE_LIMITED category=rate_limit retryable=true retryAfterMs=2000] Too many requests — try again shortly

```json
{"code":"RATE_LIMITED","category":"rate_limit","retryable":true,"retryAfterMs":2000}
```
```

With this information, the agent knows to wait 2 seconds and retry — instead of giving up or retrying immediately.

## Custom Error Codes

For domain-specific errors, construct `ToolError` directly with a custom code:

```typescript
import { ToolError } from '@opentabs-dev/plugin-sdk';

throw new ToolError('Message is too long for this channel', 'MESSAGE_TOO_LONG', {
  category: 'validation',
  retryable: false,
});
```

The `code` field is a machine-readable string. Use `UPPER_SNAKE_CASE` by convention. The `notFound` factory also accepts a custom code:

```typescript
throw ToolError.notFound('Project not found', 'PROJECT_NOT_FOUND');
```

## Best Practices

**Be specific in error messages.** The message is shown to both the AI agent and the user. Include what failed and what to do about it.

```typescript
// Good — tells the agent what to do
throw ToolError.auth('Not logged in — open app.example.com and sign in');

// Bad — vague
throw ToolError.auth('Auth failed');
```

**Use `retryAfterMs` when you know the delay.** If the API tells you when to retry (via a `Retry-After` header), pass it through. The agent uses this to wait the right amount of time.

```typescript
throw ToolError.rateLimited('Rate limited by Slack API', 3000);
```

**Prefer factory methods over the constructor.** They encode best practices (e.g., `rateLimited` is always retryable, `auth` never is). Use the constructor only when you need a custom code with a standard category.

**Let Zod handle input validation.** Your tool's Zod schema validates input before `handle()` runs. Use `ToolError.validation()` only for semantic validation that Zod can't express — like checking that a referenced resource actually exists.

**Don't catch errors you can't handle.** If an unexpected error occurs, let it propagate. The platform wraps uncaught errors in a generic error response. Only catch errors when you can provide better context through a `ToolError`.
