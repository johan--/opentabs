## Config file location

The default config file path is:

```
~/.opentabs/config.json
```

Override the config directory with the `OPENTABS_CONFIG_DIR` environment variable:

```bash
OPENTABS_CONFIG_DIR=/tmp/my-opentabs bun --hot platform/mcp-server/dist/index.js
```

The config file is created automatically on first server run with sensible defaults. The directory is created with `0700` permissions and the file with `0600` permissions.

## Example config

```json
{
  "plugins": [
    "../plugins/slack",
    "/Users/you/projects/my-opentabs-plugin",
    "opentabs-plugin-jira"
  ],
  "tools": {
    "slack_delete_message": false
  },
  "secret": "a1b2c3d4e5f6...64 hex characters..."
}
```

## Field details

### plugins

A unified array of plugin specifiers. Each entry is either a **local directory path** or an **npm package name**. The server determines the type based on the prefix:

- **Local paths** start with `./`, `../`, `/`, or `~/` — they point to a plugin directory containing `package.json` (with an `opentabs` metadata field), `dist/tools.json`, and `dist/adapter.iife.js`
- **npm package names** are everything else (e.g., `opentabs-plugin-jira`, `@myorg/opentabs-plugin-linear`) — resolved via `Bun.resolveSync` from the server's `node_modules`

**Local path resolution:**

- **Relative paths** are resolved from the config directory (`~/.opentabs/` by default)
- **Absolute paths** and `~/` paths are used as-is (with `~/` expanded to `$HOME`)
- **Security**: paths are validated with `realpath()` to resolve symlinks, then checked against `$HOME` and `$TMPDIR` — plugins outside these directories are rejected
- The `create-opentabs-plugin` scaffolder and `opentabs plugin add` command both auto-register plugins in this array

### tools

A map of prefixed tool names to booleans. This lets you disable specific tools without removing the plugin:

```json
{
  "tools": {
    "slack_delete_message": false,
    "slack_send_message": false
  }
}
```

- **Absent keys** default to enabled — you only need entries for tools you want to disable
- **Tool names** use the `<plugin>_<tool>` format (e.g., `slack_send_message`, `browser_screenshot_tab`)
- Changes are persisted atomically via a read-modify-write pattern with a mutex to prevent races

### secret

A 64-character hex string (256-bit cryptographic random) used for authentication:

- **MCP endpoint** (`/mcp`): sent as `Authorization: Bearer <secret>` header
- **WebSocket** (`/ws`): sent via the `Sec-WebSocket-Protocol` header (keeps secrets out of URL/logs)
- **Admin endpoints** (`/reload`, `/extension/reload`): require the same Bearer token

Auto-generated on first server run. If the field is missing from an existing config (e.g., after upgrading), the server generates a new secret and writes it back.

## Config directory structure

The `~/.opentabs/` directory contains more than just the config file:

```
~/.opentabs/
├── config.json              # Configuration file
├── extension/               # Managed Chrome extension install
│   ├── manifest.json        # Extension manifest
│   ├── background.js        # Service worker
│   ├── adapters/            # Plugin adapter IIFEs
│   │   ├── slack.js
│   │   └── ...
│   └── .opentabs-version    # Version marker
└── logs/                    # Server logs
```

The `extension/` directory is managed by `opentabs setup` and `bun run build`. Load this directory as an unpacked extension in `chrome://extensions`.

## Atomic writes

Config changes (from the server, CLI commands, or the `/reload` endpoint) use an atomic write pattern: content is written to a temporary file with `0600` permissions, then renamed over the target. This prevents readers from seeing partially-written files. A mutex serializes concurrent write operations.
